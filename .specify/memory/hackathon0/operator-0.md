# OPERATOR 0 — FOUNDATION LOCK FOR HACKATHON 0

## Purpose

This file defines the governing rules for applying a `.specify` layer to the Hackathon 0 repository.

This project is already implemented, already evidenced, and already prepared for submission. The `.specify` layer is being added only as a governance, clarity, and documentation layer.

It does not redefine runtime truth.
It does not rewrite project history.
It does not authorize refactoring of working code.

## Project State

Hackathon 0 is an already-built system with:

- real workflow behavior
- real logs
- real evidence artifacts
- real HITL control points
- real branch history
- real Gold and Platinum documentation

The final submission branch is:

- `platinum-final-demo`

## Operator 0 Rule

All `.specify` content in this repository must follow this rule:

> The spec layer must describe the existing system truthfully, protect runtime integrity, preserve evidence credibility, and improve judge readability without changing how the system works.

## Allowed Scope

The `.specify` layer may:

- document system purpose
- document workflow behavior
- document HITL governance
- document execution boundaries
- document evidence and traceability rules
- document Gold vs Platinum scope
- document submission branch intent
- record governance decisions
- clarify terminology

## Forbidden Scope

The `.specify` layer must not:

- rewrite runtime scripts
- rename working folders
- alter execution flow
- alter logs
- alter evidence artifacts
- backfill fake development history
- claim the system was originally built spec-first
- create misleading statements that conflict with repository evidence

## Source of Truth

The primary source of truth remains:

- actual repository structure
- actual runtime scripts
- actual logs
- actual artifacts
- actual workflow outputs
- actual branch history
- actual judge-facing documentation already present in the project

The `.specify` layer is a descriptive governance layer, not the historical source of implementation truth.

## Protected Areas

The following areas must remain protected from spec-driven modification:

- `watchers/`
- `tools/`
- `infra/`
- `ui/` unless explicitly reviewed for wording-only alignment
- `AI_Employee_Vault/Logs/`
- `AI_Employee_Vault/Artifacts/`
- `AI_Employee_Vault/Needs_Action/`
- `AI_Employee_Vault/Plans/`
- `AI_Employee_Vault/Pending_Approval/`
- `AI_Employee_Vault/Approved/`
- `AI_Employee_Vault/Done/`
- `AI_Employee_Vault/Updates/`
- historical evidence screenshots
- old branches and prior tags

## Required Tone

All spec files must be:

- truthful
- specific
- bounded
- non-hyped
- judge-readable
- consistent with real repository behavior

All spec files must avoid:

- exaggerated AI claims
- vague promises
- startup-style fluff
- fake automation narratives
- claims unsupported by evidence

## Intent of the Spec Layer

The intent of `.specify` in this repository is to:

- reduce vibe-coding perception
- make architectural intent visible
- make governance visible
- make execution boundaries visible
- make proof structure easier to understand
- strengthen submission clarity without risking system stability

## Sequencing Rule

The `.specify` layer must be developed in this order:

1. Operator 0 foundation
2. Constitution and decisions
3. Overview spec
4. Workflow and HITL governance specs
5. Execution boundary and evidence specs
6. Gold vs Platinum and submission branch specs

## Final Instruction

If any future spec content conflicts with the actual repository, logs, artifacts, or runtime behavior, the spec content must be corrected.

If forced to choose between appearance and truth, truth wins.
